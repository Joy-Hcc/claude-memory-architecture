# 设计记录：三层记忆模型的演化

> **这是设计背景参考，不是使用指南。**
> 想直接使用三层模型，读 [README.md](README.md) 的快速开始即可。
> 本文记录的是作者在 Claude Code 使用过程中，模型如何从初始状态逐步演化到最终架构。

## 从"两层查找"到"三层模型"

### 最初状态

- Memory（`raw/memory/`）：存行为调教信息
- 知识库：存用户的普通笔记
- 查询逻辑：先查 memory → 没有才查知识库

### 关键转折

**"知识库"改名为"数据库"** — 里面存的不只是"知识"，而是关于用户的所有数据。

**目录从 `knowledge` 改名为 `huochao`** — 从"知识库"变成"个人数据库"。

**"数据库就是我的记忆"** — 数据库不是外挂资料库，而是 Claude 自己的记忆。

### 最终架构（三层模型）

| 层 | 位置 | 角色 |
|---|---|---|
| memory（启动态） | `~/.claude/memory/` | 自动加载，行为指令和现场感 |
| 知识库（知识态） | 任意外部 Markdown 存储 | 持久存储，需要时主动查询 |
| 训练知识 | — | 通用兜底，编程/推理/常识 |

### 查询决策树

```
用户问题 → memory 有答案？→ 直接用
         → memory 不够？→ 翻知识库（目录 + 关键词搜索）
         → 都没有？→ 用训练知识
```

### memory 保留标准

**是不是每次对话都需要它来让工作正常进行？**
- 是 → memory（行为指令/当前上下文）
- 否 → 知识库（需要时才查阅）

### 迁移结果（2026-05-13）

- memory：11 个行为指令文件 + 1 个索引（MEMORY.md），零项目文件
- 知识库：16 个项目文件
- 所有项目索引指向知识库，memory 纯行为化

## 演进（2026-05-12）

### user-profile 分层

user-profile.md 从 95 行全量档案拆为两层：

- **memory 摘要** — 身份、沟通风格、思维模式的最小轮廓，稳定不变
- **知识库详版** — 工具路径、投资、软件版本等常变数据

原则：一份文件同时包含行为层和事实层时，memory 保留行为核心，详情放知识库。两处都由 Claude 维护，不需要用户手动同步。

### model-switching-strategy 迁移

与 Claude Code 无关的外部模型策略从 memory 移入知识库。

### feedback-firefox-migration 删除

该教训已被 collaboration-style 覆盖（先讨论再执行），且谨慎操作是通用原则，不需要单独规则。Memory 从 8 减至 7 个文件。

## 演进（2026-05-13）

### 项目索引归知识库，memory 纯行为化

- **删除所有项目缓存文件** — 8 个项目文件从 memory 移除
- **MEMORY.md 项目区全部指向知识库路径** — memory 只存路标，内容在知识库
- **存入规则补全** — 知识/项目信息不建 memory 文件，behavior 指令才建
- **auto-record-project-context 修正** — 新项目写知识库 + MEMORY.md 加索引
- **memory 从 7 → 11 个行为指令文件**

核心原则：MEMORY.md 保留所有索引（feedback + 项目路标），但项目内容文件全在知识库。索引不污染模型——路标和内容是两个东西。

## 关键洞察

1. **Memory 和知识库不是"谁的查询优先级更高"的问题，而是职责不同** — 一个是每次都需要的行为指令，一个是偶尔查阅的知识
2. **目录即索引** — 不需要单独的索引层，目录结构本身就是最好的索引
3. **路标不是内容** — MEMORY.md 里留检索起点不会污染模型，但项目详情不应该出现在 memory 中
4. **两份维护，粒度不同** — user-profile 这种跨层文件，memory 保留稳定摘要，知识库保留完整详情
