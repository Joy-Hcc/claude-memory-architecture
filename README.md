# Claude Memory Architecture — 三层记忆模型

> 让 AI 助手拥有有效的长期记忆：不是靠更大的上下文窗口，而是靠分层架构。

## 一句话理解

**自动加载的只放行为指令，需要时再查的放知识库。** 让 AI 既能"记得你是谁"，又不会被过多信息淹没。

## 三层架构

```
┌─────────────────────────────────────────┐
│       Layer 1: Memory（启动态）            │
│  自动加载到每次会话，零延迟可用              │
│  放：行为指令、偏好、沟通风格、活跃项目索引    │
├─────────────────────────────────────────┤
│       Layer 2: 知识库（知识态）              │
│  持久存储，需要时主动查询                    │
│  放：项目详情、技术文档、工具配置、踩坑记录    │
├─────────────────────────────────────────┤
│       Layer 3: 训练知识（兜底态）            │
│  模型内置的编程/推理/常识                    │
│  前两层覆盖不到时自动启用                    │
└─────────────────────────────────────────┘
```

### Layer 1: Memory（启动态）

- **位置**: `~/.claude/memory/`（Windows: `%USERPROFILE%\.claude\memory\`）
- **加载方式**: Claude Code 每次会话自动注入 `MEMORY.md` 索引的所有文件
- **判断标准**: **每次对话都需要它来让工作正常进行？** → 放 memory
- **典型内容**: 沟通风格、行为偏好、当前活跃项目索引、协作规则

### Layer 2: 知识库（知识态）

- **位置**: 任意外部存储 — Obsidian / Notion / 本地文件夹 / Git 仓库 均可
- **访问方式**: 需要时 AI 主动搜索目录 + 关键词精准定位
- **判断标准**: **需要时才查阅的具体信息？** → 放知识库
- **典型内容**: 项目技术栈、API 文档、部署配置、学习笔记、踩坑记录

> **推荐 Obsidian** — 本地 Markdown 文件，目录即索引，零依赖，和 Claude Code 协作天然契合。但模型本身不绑定任何特定工具。

### Layer 3: 训练知识（兜底态）

前两层都找不到答案时，模型用自己的训练知识（编程、推理、常识）兜底。这一层不需要你维护。

## 查询决策树

```
用户提问
    │
    ▼
Memory 有答案？──是──▶ 直接用（零延迟，不消耗推理预算）
    │
    否
    │
    ▼
翻知识库对应目录 + 关键词搜索
    │
    ▼
找到了？──是──▶ 精准读取，回答问题
    │
    否
    │
    ▼
训练知识兜底
```

## 写入决策树

```
用户说"记住这个"
    │
    ▼
每次对话都需要？──是──▶ 写 memory 文件（行为指令）
    │
    否（项目信息、技术细节、参考资料）
    │
    ▼
写知识库，然后在 MEMORY.md 加一行路径索引
```

**核心原则**: 知识/项目内容去知识库，memory 只存行为指令 + 路标（检索起点）。

## 快速开始

### 1. 创建目录结构

```bash
mkdir -p ~/.claude/memory
```

### 2. 创建 MEMORY.md（索引文件）

在 `~/.claude/memory/MEMORY.md`:

```markdown
# Memory 索引

## 行为指令（每次会话自动加载）

- [协作风格](collaboration-style.md) — 沟通方式、决策偏好
- [编码规范](coding-preferences.md) — 语言、框架偏好

## 项目（知识库路标）

- [我的博客项目](../Obsidian/projects/my-blog.md) — 技术栈、部署方式
- [工作项目](../Obsidian/projects/work-app.md) — 架构、关键决策
```

### 3. 写第一个 memory 文件

`~/.claude/memory/collaboration-style.md`:

```markdown
---
name: collaboration-style
description: 协作方式与沟通偏好
type: feedback
---

## 沟通风格

- 讨论技术方案时先给结论再展开，不要铺垫
- 做设计决策时至少给 2 个方案，说明利弊

## 编码偏好

- 主语言 TypeScript / Python
- 偏好函数式风格，避免 class
- 测试覆盖率至少 80%
```

### 4. 在 CLAUDE.md 中声明三层模型

`~/.claude/CLAUDE.md`（或项目级 `.claude/CLAUDE.md`）:

```markdown
# 记忆系统

本项目使用三层记忆模型：
1. Memory（`~/.claude/memory/`）— 行为指令，自动加载
2. 知识库（`~/Obsidian/` 或你的知识库路径）— 项目详情，需要时查询
3. 训练知识 — 通用兜底

## 查询规则

- 用户提问后，先检查 memory 中是否有答案
- memory 不够时，搜索知识库对应目录
- 知识/项目信息不建 memory 文件，写入知识库 + MEMORY.md 加路径索引
- 行为指令类信息写入 memory 文件
```

### 5. 开始使用

现在每次打开 Claude Code，memory 文件会自动注入。遇到项目相关问题，AI 会主动搜索你的知识库。

## Memory 文件类型

Claude Code 的 memory 支持四种类型：

| 类型 | 用途 | 示例 |
|------|------|------|
| `user` | 用户角色、偏好、知识背景 | "我是全栈开发者，主语言 TS" |
| `feedback` | 行为规则、协作偏好 | "不要过度工程化" |
| `project` | 项目上下文（路标型，内容在知识库） | "博客项目技术栈见 Obsidian" |
| `reference` | 外部资源定位 | "CI 配置在 Jenkins 某某地址" |

## 常见问题

### Memory 文件越多越好吗？

不是。Memory 每次会话全量加载，文件越多消耗的上下文越多。**控制在 20 个以内**，只放真正"每次都需要"的行为指令。

### 知识库用什么工具？

任意 Markdown 文件夹即可。推荐 Obsidian（免费、本地优先、目录即索引），但 Notion、Logseq、甚至一个 Git 仓库的 `docs/` 目录都可以。

### 怎么知道一个东西该放 Memory 还是知识库？

问自己：**"下次打开 Claude Code 时，它不需要知道这个也能正常工作吗？"**
- 需要 → Memory
- 不需要，但以后可能会问到 → 知识库

### 项目文件为什么不能放 Memory？

项目详情（技术栈、架构、配置）是"偶尔查阅"的信息，不是"每次都需要"的行为指令。放 Memory 会白白消耗上下文窗口，而放知识库可以按需精准检索。

## 适用场景

- Claude Code + 任意 Markdown 知识库（Obsidian / Notion / 本地文件夹）
- 任何需要在多次对话间保持上下文的 AI 助手
- 个人知识管理与 AI 辅助工作的结合

核心思想可以推广到任何 LLM 应用：**自动加载行为，按需查询知识。**

## 设计背景

这个模型是在 Claude Code 使用过程中逐步演化出来的。原始问题：memory 和知识库双重写入，内容分叉，查询路径混乱 → 经过多轮迭代收敛到三层各司其职。
