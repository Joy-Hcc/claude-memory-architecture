# Claude Memory Architecture — 三层记忆模型

> 如何让 AI 助手拥有有效的长期记忆？不是靠更大的上下文窗口，而是靠分层架构。

## 背景

Claude Code 有一个原生记忆系统：通过 `MEMORY.md` 索引自动加载 memory 文件到每次会话。同时我们维护了一个 Obsidian 知识库作为个人数据库。

**最初的问题：** 两个系统同时存一样的东西——项目信息、工具配置、参考资料在两边各有一份。双重维护，内容分叉，查询路径混乱。

经过多轮讨论和迭代，我们设计出了三层模型。

## 三层架构

```
┌─────────────────────────────────────┐
│         Layer 1: Memory（启动态）      │
│    自动加载，行为指令和现场感             │
│    每次会话零延迟可用                    │
├─────────────────────────────────────┤
│         Layer 2: Obsidian（知识态）     │
│    持久存储，需要时主动查询               │
│    目录即索引，精准定位                   │
├─────────────────────────────────────┤
│         Layer 3: 训练知识（兜底态）      │
│    通用编程/推理/常识                    │
│    前两层都覆盖不到时使用                 │
└─────────────────────────────────────┘
```

### Layer 1: Memory（启动态）

- **位置**: `~/.claude/memory/` 或项目 `.claude/memory/`
- **加载方式**: 每次会话系统自动注入
- **存储内容**: 纯行为指令和现场感——"一醒来就需要知道"的东西
- **判断标准**: **是不是每次对话都需要它来让工作正常进行？** 是 → memory

示例：用户偏好、沟通风格、行为规则、当前活跃项目列表

### Layer 2: 知识库（知识态）

- **位置**: 任意持久化存储（Obsidian、文件系统、数据库均可）
- **访问方式**: 需要时主动搜索——目录 + 关键词精准定位
- **存储内容**: 项目详情、工具配置、方法论、参考资料
- **判断标准**: **需要时才查阅的具体信息？** 是 → 知识库

示例：项目技术栈和架构、API 配置格式、工具安装步骤、踩坑记录

### Layer 3: 训练知识（兜底态）

- **位置**: 模型参数
- **访问方式**: 零延迟内检索
- **作用**: 编程、推理、常识——前两层覆盖不到时自动启用

## 查询决策树

```
用户提问
    │
    ▼
Memory 有答案？──是──▶ 直接用（零延迟）
    │
    否
    │
    ▼
翻知识库对应目录 + 关键词搜索
    │
    ▼
找到了？──是──▶ 精准读取，回答问题
    │
    否
    │
    ▼
训练知识兜底
```

**关键洞察:** 翻知识库不是因为 Memory 不够，而是因为有些东西本来就该在知识库里。Memory 是行动态，知识库是知识态，各司其职。

## 写入原则

- 用户说"记住这个" → 判断属于行为指令（存 Memory）还是知识（存知识库）
- 不写格式化的信息卡片，不建复杂的分类体系
- 目录结构本身就是索引，不需要单独的索引层
- 尽量少维护，让信息自然沉淀

## 实际效果

迁移前：Memory 23 个文件，大量项目信息和工具配置重复存储在两个系统

迁移后：Memory 9 个文件（纯行为指令），35+ 文件在知识库（项目/工具/方法/参考）

## 适用场景

这个模型适用于：
- Claude Code + 任意知识库（Obsidian、Notion、本地文件系统）
- 任何需要在多次对话间保持上下文的 AI 助手
- 个人知识管理和 AI 辅助工作的结合

核心思想可以推广：**自动加载存行为指令，需要时再查存知识——让 AI 既能"记得你是谁"，又不会被过多信息淹没。**

## 设计讨论记录

整个架构是在 Claude Code 对话中逐步演化出来的：

1. 发现双重写入问题 → 重新审视"数据库就是记忆"的理念
2. 尝试 A 方案（Memory 纯索引）→ 发现查询成本高、和现状差别小
3. 回到 B 方案（Memory 存精要 + 指针）→ 实际是找到光谱而非开关
4. 最终收敛：用户提出"目录即索引"——消除二元对立，三层各司其职
